## should the pact file need to be generate on consumer?
Yes, the Pact file must be generated by the consumer during the contract testing process. Here’s the typical workflow:

### 1. Consumer Side: Generate the Pact File
- On the consumer side, you write tests using the Pact framework (e.g., PactDslJsonBody for creating expectations).
- When the consumer tests run, they generate a Pact file. This file captures the expected requests and responses between the consumer and the provider.

The Pact file is usually saved to a location like:

````bash
src/test/resources/pacts/ConsumerService-ProviderService.json
````

###  Example: Consumer Pact Test (JUnit 5)
````java
import au.com.dius.pact.consumer.dsl.PactDslWithProvider;
import au.com.dius.pact.consumer.junit5.PactConsumerTestExt;
import au.com.dius.pact.consumer.junit5.PactTestFor;
import au.com.dius.pact.core.model.RequestResponsePact;
import org.junit.jupiter.api.extension.ExtendWith;
import static org.assertj.core.api.Assertions.assertThat;

@ExtendWith(PactConsumerTestExt.class)
public class ConsumerPactTest {

    @PactTestFor(providerName = "ProviderService", port = "8080")
    RequestResponsePact createPact(PactDslWithProvider builder) {
        return builder
            .given("user exists")
            .uponReceiving("GET REQUEST")
            .path("/api/pact")
            .method("GET")
            .willRespondWith()
            .status(200)
            .body("{\"condition\": true, \"name\": \"tom\"}")
            .toPact();
    }

    @Test
    void testConsumer() {
        // Code to test the consumer using the mock provider
        String response = new HttpClient().get("http://localhost:8080/api/pact");
        assertThat(response).isEqualTo("{\"condition\": true, \"name\": \"tom\"}");
    }
}

````
Running this test will generate the Pact file

### 2. Commit the Pact File:
After generating the Pact file, it should be committed to your source code repository so it can be used by the provider for verification.

### 3. Provider Side: Verify the Pact File
On the provider side, you use the Pact file to verify that the provider service behaves according to the expectations defined by the consumer.

````java
import au.com.dius.pact.provider.junit5.PactVerificationContext;
import au.com.dius.pact.provider.junit5.PactVerificationInvocationContextProvider;
import au.com.dius.pact.provider.junit5.HttpTestTarget;
import au.com.dius.pact.provider.junit5.PactFolder;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.TestTemplate;
import org.junit.jupiter.api.extension.ExtendWith;
import org.springframework.boot.test.context.SpringBootTest;

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.DEFINED_PORT)
@PactFolder("src/test/resources/pacts")// Pointing to the folder where the Pact file is stored
public class ProviderPactTest {

    @BeforeEach
    void setup(PactVerificationContext context) {
        context.setTarget(new HttpTestTarget("localhost", 8080));
    }

    @TestTemplate
    @ExtendWith(PactVerificationInvocationContextProvider.class)
    void verifyPact(PactVerificationContext context) {
        context.verifyInteraction();
    }
}

````

### 4. Upload the Pact File to a Pact Broker (Optional but Recommended)
To facilitate collaboration between consumer and provider teams, you can upload the Pact file to a Pact Broker (e.g., `arpangroup.pactflow.io`). This allows providers to easily access and verify the Pact file.
````bash
pact-broker publish src/test/resources/pacts --broker-base-url https://arpangroup.pactflow.io --broker-token <your-token>
````

## Summary:
1. Consumer: Generates the Pact file based on consumer tests.
2. Commit: Pact file is committed and shared with the provider team.
3. Provider: Uses the Pact file to verify the API implementation.
4. Pact Broker (Optional): Centralizes the Pact files for better collaboration.


## what the use of this mockServerUrl?
The `mockServerUrl` is used in Pact testing to provide the URL of the mock server that simulates the provider's API. *When you're writing a consumer contract test, the idea is that you're testing how the consumer interacts with the provider,* but instead of making requests to the actual provider (which might be unavailable or unreliable during testing), you use a mock server to simulate the provider's behavior.
